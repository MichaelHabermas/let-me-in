Background
Physical access control is a multi-billion-dollar industry undergoing rapid transformation. Companies like
Verkada, ButterflyMX, and Openpath have replaced legacy key-card systems with camera-first, AI-powered
entry platforms. Apple's Face ID normalized facial biometric authentication for consumers, while AWS
Rekognition and Azure Face API commoditized cloud-based face matching. The core value proposition is
simple: eliminate badges, PINs, and tailgating by verifying identity through what a person is rather than what
they carry.
You will build Gatekeeper — a fully browser-based door-entry verification system. The application captures a
live webcam frame of a person attempting entry, runs YOLOv9 face detection locally in the browser, extracts
facial features, and compares them against a pre-enrolled database of authorized personnel. The critical
technical challenge is running the entire inference pipeline client-side using ONNX Runtime Web or
TensorFlow.js — no server-side GPU, no cloud API calls for inference. This forces you to grapple with model
optimization, quantization tradeoffs, and real-time performance on constrained hardware.

Deep Dive: Browser-Based YOLOv9 Face Recognition
Pipeline
This is the project's signature technical challenge. You are running a full object-detection model in the browser
and building a face-matching pipeline on top of it — no Python backend, no cloud inference. Everything
happens in JavaScript with WebGL or WASM acceleration.

Required Capabilities
Your system must handle these scenarios correctly:

• Model Loading: Load a YOLOv9 ONNX model via onnxruntime-web. Display a loading progress bar.
Handle model download failures gracefully.
• Live Detection: Continuously capture frames from the webcam, run YOLOv9 inference, and draw
bounding boxes around detected faces in real-time on a canvas overlay.
• Embedding Generation: After detecting a face, crop the region, preprocess it (resize, normalize), and
generate a fixed-length embedding vector for comparison.
• 1:N Matching: Compare the live embedding against all enrolled embeddings. Return the best match
and its similarity score. Handle the case where no match exceeds the threshold.
• Enrollment Capture: Provide a guided enrollment flow: 'Look at the camera ® Detecting face ®
Captured ® Enter details ® Saved.'

Model Pipeline Architecture
Your pipeline has three stages. Each must be independently testable:

 Stage           Input                 Output                       Technology

 1. Detection    Raw webcam frame (640×480)
                                       Bounding box coordinates + confidence
                                                                    YOLOv9 ONNX model

 2. Extraction   Cropped face region   Embedding vector (128-d or 512-d)
                                                                    FaceNet/ArcFace ONNX or custom head

 3. Matching     Live embedding + enrolled Best
                                       DB     match ID + similarity score Cosine similarity in JS


ONNX Model Integration Schema
Your model loading and inference code should follow this pattern:

 // Model initialization
 const session = await ort.InferenceSession.create('./models/yolov9-face.onnx', {
 executionProviders: ['webgl', 'wasm'], // WebGL preferred, WASM fallback
 graphOptimizationLevel: 'all'
 });
 // Preprocessing: resize frame to 640x640, normalize to [0,1]
 const tensor = new ort.Tensor('float32', preprocessed, [1, 3, 640, 640]);
 // Run inference
 const results = await session.run({ images: tensor });
 const detections = postprocess(results); // NMS, threshold filtering

Evaluation Criteria

     Test Input Expected Output

     Enrolled user facing camera, good lighting GRANTED + correct name + score ³ 0.80

     Enrolled user, low lighting / partial occlusion GRANTED or UNCERTAIN + score ³ 0.65

     Unenrolled person facing camera DENIED + 'Unknown' + score < 0.60

     Printed photo held to camera DENIED or flagged as potential spoof (stretch goal)

     Empty frame (no face) No detection — 'No face detected' message displayed

     Two faces in frame Prompt: 'Multiple faces detected. Please ensure only one person is in frame.'

Build Strategy


Priority Order
1. YOLOv9 model conversion and browser loading — this is the hardest unknown. Convert the model to
ONNX, verify it loads in onnxruntime-web, and confirm inference produces valid outputs. Do this first.
2. Webcam capture pipeline — get getUserMedia working, render frames to a canvas, extract frame data as
tensors.
3. Face detection integration — feed webcam frames to the loaded model, parse bounding box outputs,
draw overlays.
4. Embedding extraction — implement or integrate a face embedding model to generate feature vectors
from detected faces.
5. Enrollment system — build the admin flow to capture a face, generate its embedding, and store it in
IndexedDB.
6. Matching engine — implement cosine similarity search across enrolled embeddings; apply threshold logic.
7. Access decision UI — wire up the GRANTED/DENIED display, side-by-side comparison, and entry
logging.
8. Polish and stretch features — liveness detection, scheduling, audit export, sound feedback.

Critical Guidance
    • Start with a pre-converted ONNX model. Do not attempt to train YOLOv9 from scratch — find a
    pre-trained face detection checkpoint and convert it. The conversion alone may take hours to debug.
    • Test ONNX Runtime Web on day one. If WebGL inference fails on your hardware, you need to know
    immediately so you can fall back to WASM or swap model architectures.
    • Keep your face database small during development — 3-5 enrolled users. Scaling to 50 is an
    optimization problem, not a functionality problem.
    • Use cosine similarity, not Euclidean distance, for embedding comparison. Cosine similarity is
    scale-invariant and works better with normalized embeddings.
    • Profile your pipeline early. If detection takes 2 seconds per frame, you cannot brute-force your way to
    500ms. You need model quantization or a smaller model.

     • Deploy to Netlify or Vercel by end of day 1. Having a live URL forces you to test on real devices and
     reveals CORS, HTTPS, and model-loading issues early.

Phase 1: Define Your Constraints

1. Scale & Load
    • How many users will be enrolled in the face database at launch? At maximum capacity?
    • How many verification attempts per minute does the system need to handle?
    • What is the maximum acceptable latency from camera capture to access decision?
    • Will multiple entry points (cameras) operate simultaneously against the same database?

2. Budget & Resources
    • What is your budget for AI/ML development tools this week (API costs, compute)?
    • Do you have access to a GPU for model conversion and quantization, or are you CPU-only?
    • What are the target deployment costs per month for the production application?
    • Can you use free-tier hosting, or do you need paid infrastructure for model file serving?

3. Timeline & Milestones
    • What is the absolute minimum feature set for a working demo in 24 hours?
    • Which parts of the pipeline have you never built before and carry the most time risk?
    • What is your fallback plan if YOLOv9 ONNX conversion fails — do you have a backup model?
    • How much time will you allocate to model conversion vs. application development?

4. Privacy & Compliance
    • Where are face images and embeddings stored? What are the privacy implications of client-side
    storage?
    • Does your target deployment region have facial recognition regulations (e.g., Illinois BIPA, EU AI Act)?
    • How will you handle data deletion requests? Can users request their face data be removed?
    • Will you store raw face images, or only embeddings? What is the re-identification risk of stored
    embeddings?

5. Team & Skills
    • Have you worked with ONNX Runtime Web or TensorFlow.js before? What is your experience level?
    • Are you comfortable with canvas-based image manipulation and WebGL concepts?
    • Do you have experience with face detection or recognition models? Which ones?

Phase 2: Architecture Discovery

1. Model Selection & Conversion

    • Which YOLOv9 variant (tiny/small/medium) best balances accuracy and browser inference speed?
    • What is the exact conversion path: PyTorch ® ONNX ® quantized ONNX? What tools do you need
    (ultralytics, onnx, onnxruntime)?
    • What input resolution will you use (320×320, 640×640)? How does this affect detection accuracy and
    speed?
    • Have you verified that your chosen ONNX model works with onnxruntime-web? Are there known
    operator compatibility issues?
    • Will you use INT8 or FP16 quantization? What is the expected accuracy vs. speed tradeoff?

2. Face Embedding Strategy
    • Will you use a separate face embedding model (FaceNet, ArcFace) or attempt to use YOLOv9 features
    directly?
    • What embedding dimensionality (128-d, 256-d, 512-d) gives the best tradeoff for your use case?
    • Is there a pre-converted ONNX embedding model available, or do you need to convert one yourself?
    • How will you normalize embeddings before similarity comparison?

3. Browser ML Runtime
    • ONNX Runtime Web vs. TensorFlow.js — which has better support for your model architecture?
    • What execution providers are available (WebGL, WebGPU, WASM)? Which does your target browser
    support?
    • How will you handle model loading on slow connections? Progressive loading? Web Workers?
    • What is the memory ceiling for a browser tab on your target devices?
    • Can you run detection and embedding extraction concurrently using Web Workers?

4. Client-Side Storage Architecture
    • IndexedDB schema design: what object stores do you need (users, embeddings, logs, settings)?
    • How will you handle IndexedDB versioning and schema migrations?
    • What is the maximum practical size of an IndexedDB database on your target browsers?
    • Will you use a library like Dexie.js or raw IndexedDB API? What are the tradeoffs?

5. Webcam & Image Processing
    • What resolution will you request from getUserMedia? How does this interact with model input size?
    • How will you handle camera permissions — prompt timing, denial handling, device switching?
    • What preprocessing steps are needed: color space conversion, normalization, tensor formatting?
    • How will you handle aspect ratio mismatches between webcam output and model input?

6. Matching & Decision Logic
    • Cosine similarity vs. Euclidean distance vs. L2 normalized distance — which will you use and why?
    • How will you determine the optimal similarity threshold? Will you use a validation set?
    • How will you handle the case where multiple enrolled users exceed the threshold for a single face?
    • What is your strategy for updating enrolled embeddings over time as a person's appearance changes?

Phase 3: Post-Stack Refinement

1. Security & Failure Modes
    • How will you prevent spoofing via printed photos or video replay? What liveness detection is feasible
    in-browser?
    • What happens if the ONNX model fails to load? What is the graceful degradation path?
    • How will you secure the admin enrollment interface — is it password-protected?
    • What are the risks of storing facial embeddings in IndexedDB (accessible via browser DevTools)?

2. Testing Strategy
    • How will you test face matching accuracy? Will you create a benchmark dataset with known-good pairs?
    • How will you test under different lighting conditions, distances, and angles?
    • What browser compatibility testing will you do (Chrome, Firefox, Safari, mobile)?

3. Deployment & Hosting

    • Does your deployment platform support the required CORS headers for ONNX model loading?
    • How will you handle HTTPS requirements for getUserMedia on deployed environments?
    • What is your rollback plan if a deployment breaks model loading?

4. Observability & Debugging
    • How will you log inference performance metrics (detection time, embedding time, match time) in
    production?
    • What visual debugging tools will you build — bounding box overlays, confidence scores, embedding

    • How will you diagnose false positives and false negatives? Can you capture and review failed matches?

5. Performance Optimization
    • If detection is too slow, what is your model downsizing strategy (smaller variant, lower resolution, fewer
    anchors)?
    • Can you skip frames for detection while maintaining a smooth webcam preview?
    • What caching strategies will you use for model inference sessions and preprocessing buffers?
    • How will you profile memory usage to prevent browser tab crashes on low-end devices?
