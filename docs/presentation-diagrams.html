Gatekeeper (Let Me In)

Architecture, user flows, and application runtime — for panel presentations.

Elevator pitch

  • Gatekeeper is a browser-only facial recognition access demo
  • The gate page uses the webcam; face detection runs in a Web Worker, recognition uses local embeddings compared to an on-device roster.
  • Outcomes are explicit GRANTED, UNCERTAIN, or DENIED — not a forced binary when the match is weak.
  • Admins enroll people and tune thresholds on the admin page.
  • IndexedDB on that device holds the roster (users, face embeddings, optional reference images), settings (thresholds, cooldowns), and access logs.

Talking points

  1. Three pages, one product: gate (live check), admin (enroll + tune), log (local history). No server for video or inference.
  2. Privacy: frames and embeddings stay in-browser; roster and logs live in IndexedDB (gatekeeper).
  3. ML: YOLO detection in a Web Worker; ONNX Runtime for embedding; models served as static assets.
  4. Policy: decideFromMatch maps similarity + margin to GRANTED / UNCERTAIN / DENIED.
  5. Bootstrap: secure context → DB init → mount; avoids half-loaded UI.

Talking points

  1. Visitor: browser + camera permission; continuous feedback; tri-state outcome with UNCERTAIN as a safety valve.
  2. Admin: enroll identities, persist locally, tune sensitivity for the environment.
  3. Log: device-local audit trail; central logging would need an explicit backend.
  4. Trust boundary: admin auth is front-door UI credentials (prod: env at build time) — not enterprise IAM.

3. Application flow (per frame)

Technical sequence from one camera frame to a decision and optional log write.

sequenceDiagram
  participant Cam as Camera / frame
  participant Pipe as Detection pipeline
  participant Wk as YOLO Web Worker
  participant Emb as Face embedder (ORT)
  participant Eng as Access decision engine
  participant Dom as Match + policy
  participant IDB as IndexedDB
  participant UI as Gate UI + log

  Cam->>Pipe: Video frame (ImageData)
  Pipe->>Wk: Detect faces (bbox)
  Wk-->>Pipe: Detections

  alt No face / multiple faces
    Pipe->>UI: Status + overlay only
  else Single face
    Pipe->>Emb: Crop + embed
    Emb-->>Pipe: Float32 embedding
    Pipe->>Eng: Evaluate access
    Eng->>IDB: Load users + embeddings
    IDB-->>Eng: Roster
    Eng->>Dom: Best match + margin
    Dom-->>Eng: GRANTED / UNCERTAIN / DENIED
    Eng-->>Pipe: Verdict + snapshot blob
    Pipe->>UI: Banner / meter / colors
    opt GRANTED or DENIED
      Pipe->>IDB: appendAccessLog
      IDB-->>UI: Persisted row
    end
  end

Talking points

  1. Detect first: YOLO finds faces off the UI thread; boxes drive overlay and crop.
  2. Cardinality: zero or multiple faces typically skip grant path; single face runs embedding + policy.
  3. Compare: embedding vs stored vectors (similarity + runner-up margin).
  4. Log: GRANTED and DENIED append to accessLog with frame blob; UNCERTAIN is not logged the same way (align with code when presenting).
  5. Loop: repeats at camera cadence (with debounce/cooldown) for a live-door feel.
